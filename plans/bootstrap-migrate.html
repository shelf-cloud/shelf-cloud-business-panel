2. Migration Principles

Do

  • Keep Bootstrap and Tailwind running side by side until each workflow is migrated and verified.
  • Use shadcn components as source components, not third-party black boxes.
  • Keep current data fetching, mutations, exports, auth, routing, and business logic unchanged.
  • Use Tailwind semantic tokens like tw:bg-background, tw:text-foreground, and tw:border-border.
  • Use cn() for conditional classes.
  • Keep tw: prefix until Bootstrap is fully removed.
  • Migrate shared components before page-level one-off markup.

Do not

  • Do not remove bootstrap or reactstrap before every import is gone.
  • Do not rewrite API calls or domain state while migrating UI.
  • Do not mix unprefixed Tailwind utilities with Bootstrap utility names during the transition.
  • Do not replace table behavior until sorting, pagination, expansion, selection, and loading states are covered.
  • Do not add a separate dark mode provider until current layout attributes are audited.
  • Do not manually restyle shadcn components with raw colors where variants or tokens exist.

Why this is the right end state: the tw: prefix exists only to avoid class collisions with Bootstrap, and the self-contained migration-ui wrappers exist only because tailwind-merge is not tw:-prefix aware (so shadcn defaults can't be overridden via className). Both problems vanish once Bootstrap is gone.

Known gaps: --radius maps to --bs-border-radius (0.25rem) but cards use 0.75rem (shadcn single-radius scale vs Velzon mixed radii); shadcn structure (Card py-6/gap-6) still needs polish vs Velzon (1rem + dividers).
  • Spacing scale mismatch to resolve: Velzon uses Bootstrap's scale (m-3=1rem); Tailwind uses 0.25rem base (p-4=1rem). Adopt Tailwind-native values during migration, or remap Tailwind's spacing theme.
  • Audit remaining shadcn-backed wrappers (dropdown, modal, sheet, input, label) against their Bootstrap originals — same drift risk as the card.

5. Risk Log

High risk: app shell. Sidebar, topbar, layout width, mobile overlay, and route active state are currently driven by Bootstrap-era SCSS and document attributes. Migrate late.
High risk: tables. 55 files depend on react-data-table-component behavior. Restyle first; replace engine later only if necessary.
Medium risk: class collisions. Bootstrap and Tailwind share common utility names. Keep tw: prefix until Bootstrap is removed.
Medium risk: forms. Current forms use Bootstrap validation props. Preserve validation state with shadcn Field and aria-invalid.
Medium risk: third-party plugin CSS. Flatpickr, Select, ApexCharts, Dropzone, DataTable, SimpleBar, and toast styles may rely on Bootstrap variables or old SCSS.
Low risk: RTL (already disabled). The RTL SCSS layer is inactive — all four imports are commented out in themes.scss and nothing sets dir. It cannot silently regress because it is not active. Treat as cleanup (delete the dead files) unless RTL is deliberately revived.
Medium risk: no automated regression net. No jest/vitest/playwright/cypress config and no CI workflow exist. Acceptance currently rests on lint, build, and manual eyeballing. Add Playwright snapshot coverage of top workflows before Phase 3, or visual regressions ship unnoticed.
Low risk: bundle bloat during transition. Running Bootstrap + reactstrap + Tailwind + shadcn together ships all four. Set a bundle-size budget check so the side-by-side window does not quietly balloon payload.
